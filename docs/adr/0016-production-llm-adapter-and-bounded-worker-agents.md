# The production LLM adapter: DeepSeek behind an agent-capable port, agents as bounded workers

Until now no production `LLMPort` existed — runs were driven by a scripted fake, so the admin generation-trigger shipped dormant ([ADR 0015](0015-one-backend-deployable-http-api.md)). This ADR settles how the first real adapter is built: **DeepSeek behind the existing port**, with the port widened just enough to let a stage delegate *bounded* local decisions to a `create_agent` loop — without surrendering the determinism guarantees of [ADR 0010](0010-content-generation-pipeline-architecture.md).

## Decisions

1. **DeepSeek is an `LLMPort` adapter; the provider is a config choice.** `LLM_PROVIDER=deepseek` selects it through a small registry in `harness/adapters/llm_factory.py`; a new `src/harness/config.py` (`LLMConfig.from_env`, the same pattern the store configs use) reads credentials and the model/temperature knobs. Changing model, temperature, or key is pure environment — no code change. Adding a *provider* is a new adapter class + one registry line + flipping `LLM_PROVIDER`.

2. **The port gains one method: `run_agent`.** Alongside `complete(request) -> str`, the port gets `run_agent(request, tools, *, step_limit) -> str`, where `tools` are domain-neutral `ToolSpec`s (name, description, JSON schema, a `(args) -> str` callable). `create_agent`, `ChatDeepSeek`, tool-binding, and `recursion_limit` live **entirely behind the adapter**; `stages.py` sees only `ToolSpec` and gets back JSON, decoded by the same `decode.py` as every other call. The hexagon holds — the domain never imports LangChain.

3. **Agents are bounded workers inside deterministic stages — not a supervisor.** Two stages delegate local decisions to a `run_agent` loop: the **Researcher** (Source), whose `fetch` tool lets it navigate cited outlinks *toward* primary sources instead of chasing them blindly ([ADR 0011](0011-harness-tools-are-ports-web-sourcing-recall-first.md)); and the **Editor** (Edit), whose `check_guardrails` tool (wrapping `evaluate_piece` + the judge) lets it revise against a *deterministic* checker. In both, **the stage's deterministic post-checks stay the arbiter** — the corroboration bar, `ThinSourcePackError`, `evaluate_piece() == clean` — and the model's freedom is capped by `step_limit`. **The agent is bound *first* by the expert-authored markdown**: its system prompt is the assembled Editorial DNA / Voice Profile / guardrail spec carried in `LLMRequest.instructions`, exactly as every `complete()` call, so the loop reasons with editorial taste rather than optimizing blindly against a checker or raw pages — `step_limit` and the post-checks bound an already-taste-constrained agent, not a generic tool-loop. (The Researcher's navigation is a *new* authored decision-point — a "Navigation" section added to `guardrails/sourcing.md` — because the citation-chase it replaces was pure code with no authored instructions.) The agent *proposes*; code *disposes*. This refines how [ADR 0010](0010-content-generation-pipeline-architecture.md)'s stages work internally; it does **not** reopen its shape — plan-first, the outcome contract, the human gates, the dual-runtime split, and determinism-by-construction all hold, because the agent lives behind the shared port, not in either runtime's orchestration. The Writer (Draft) and Reviewer (QA) stay single-shot `complete()` calls: an author must not grade its own draft, and a verdict stage must not loop on its own judgement.

4. **Structured output is JSON mode + an adapter-side repair/transport retry.** The adapter requests `response_format=json_object` and retries on transport errors and non-parseable output; `decode.py` stays the single authority on *shape*, failing loud on a genuine contract mismatch. This is provider-agnostic (every OpenAI-compatible provider supports JSON mode) and avoids maintaining a parallel schema per purpose.

5. **DeepSeek V4 in two tiers, chosen by request purpose.** A *tier* is a `(model, temperature)` pair: **V4 Flash at near-0** (*precise*) for the structural and judging purposes, **V4 Pro at a higher temperature** (*creative*) for prose. Both model-ids are environment config (decision 1), so re-tiering or bumping a model is a config change, never a code one; the two agent loops (decision 3) inherit their stage's tier — the Researcher's navigation is precise, the Editor's revision is creative. Whichever model a tier names must support JSON mode (decision 4), function calling (decision 2's `run_agent`), and honour `temperature` — a verify-at-build check, not a V1 assumption. This supersedes the deprecated `deepseek-chat`; V4 collapses the old `deepseek-chat`/`deepseek-reasoner` split, so that framing no longer applies.

6. **Bounded concurrency over the per-Piece fan-out.** Source, Draft, and Edit process Pieces through a `ThreadPoolExecutor` (single bound `N`, a within-stage barrier so the deliverable-on-disk gate and resume idempotence hold). Because the Playwright adapter launches a browser per fetch and its sync API is not freely thread-safe, fetching moves to a **thread-local browser** (one per worker thread, reused), bounding concurrent Chromium to the pool.

7. **Failures collect and escalate into the existing learning loop.** A concurrent fan-out stage no longer aborts on the first Piece to fail its bar; it lets every Piece finish, persists each failing artifact as the machine copy plus its failure code, and routes the failed Pieces into the **piece gate** as review targets. The human's fix is then an ordinary `edit_approve` verdict (diff-by-preservation, [ADR 0013](0013-human-review-surface-is-the-file-workspace.md)) that the Distiller already consumes ([ADR 0004](0004-human-ratified-learning-loop.md)); an unfixable Piece becomes a non-survivor and rewire → reqa re-validate the structural invariants over the survivors ([ADR 0012](0012-publish-gate-rewire-and-reqa-approved-subset.md)). No new learning machinery.

## Why

- **The moat is unchanged by the model.** Anti-slop, grounding, and structural coherence stay enforced by construction — the agents only automate the *manual* parts (navigating to a source, revising against a rule) a human would otherwise do by hand, and the deterministic guardrails still decide what is acceptable.
- **One seam, many providers.** With LangChain confined behind the port, swapping DeepSeek for another model is an adapter, not a pipeline rewrite.
- **Fail loud, but informatively.** Collect-all shows the human every failure in one pass and turns each fix into the richest learning signal there is — a failing-draft → human diff.

## Trade-off

Heavier than a single stateless `complete()` adapter: an agent loop, a tool abstraction, thread-safe concurrency, and a reworked web adapter. Accepted — the agentic stages are where the model does work a human otherwise would, and concurrency is what makes a several-hundred-call run tolerable. The cost is real DeepSeek spend and a larger surface to test; a per-run `usage.json` and a recorded-HTTP contract test bound both.

## Consequence

The [ADR 0006](0006-generation-and-consumption-are-separate.md) boundary is untouched — none of this crosses into consumption. [ADR 0010](0010-content-generation-pipeline-architecture.md) still governs the pipeline's *shape*; this ADR governs how a stage may internally delegate bounded decisions and how the first real model is wired. The dormant admin trigger ([ADR 0015](0015-one-backend-deployable-http-api.md)) goes live once `LLM_PROVIDER` and a key are set.

## Implementation note — where Decision 7's routing applies (reconciled 2026-07-06)

Decision 7's "route the failed Piece into the piece gate" is realised at the **Edit** tier, because only there does a best-effort artifact exist to review: an Edit-bar failure (`QABudgetExceededError` / `GroundingDriftError`) persists the Piece's draft as its `piece.md` machine copy plus a `failure.md` marker, and the Piece flows to the piece gate as an ordinary review target — the human's `edit_approve` fix (or a rejection → non-survivor via rewire/reqa) closes it exactly as Decision 7 describes.

A **Source** failure is different: a Piece whose vetted claim pack is too thin (`ThinSourcePackError`) has *no* draft to route and cannot pass through the full-plan Wire/QA stages that precede the piece gate. So the Source stage still **collects** — it finishes researching every Piece before failing, and reports all thin Pieces together in one aggregate error — but the run is **fatal** at research (the Piece "dies at research, not in prose"), not routed onward. The routing-to-gate path and the collect-then-abort path are the two faces of Decision 7; which one a failure takes depends on whether a reviewable draft exists.
