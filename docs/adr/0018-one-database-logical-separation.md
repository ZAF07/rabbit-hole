# Both subsystems share one Postgres database: the separation is logical, not physical

[ADR 0015](0015-one-backend-deployable-http-api.md) already settled the backend as one deployable over "one Postgres," but the connection config drifted: each subsystem read its own DSN env var (`CONTENT_GRAPH_DSN`, `CONSUMPTION_DSN`), so local setup stood up **two** physical databases to fill them. This ADR closes that drift. **Both subsystems address one Postgres database through a single `DATABASE_URL`.** Generation writes the Content Graph tables (`pieces`, `blocks`, `topics`, `connections`, …); consumption owns the reader tables (`users`, `sessions`, `journeys`, …); they coexist in one database, separated by **module and schema ownership**, never a physical split. The app auto-migrates both schemas on boot.

**Why:**
- **Nothing required two databases.** The content tables and reader tables have disjoint names, and each subsystem's forward-only migration runner records into `schema_migrations` keyed by distinct filenames — one database holds every table and every ledger row with no collision. The second database was ceremony, not architecture.
- **The boundary that matters is already enforced elsewhere.** [ADR 0006](0006-generation-and-consumption-are-separate.md)'s separation lives in **import direction and the port surface** ([ADR 0014](0014-harness-code-lives-in-src-harness.md)): consumption reads content only through `ContentGraphRepository`, generation never touches user/session state, and neither module imports the other. That holds identically whether the tables sit in one database or two — physical co-location cannot re-fuse a boundary that is a code boundary.
- **One `DATABASE_URL` removes a whole failure mode.** Two vars invited an operator to set them to two different databases and get a silently half-migrated system. One var is impossible to split by accident, and it is exactly what a managed Postgres (Supabase) hands you.
- **Simplicity before the first user** ([ADR 0015](0015-one-backend-deployable-http-api.md)): one database, one connection string, one cold-start command.

**Consequence:**
- **Config surface collapses to `DATABASE_URL`.** Both `ContentGraphConfig` and `ConsumptionConfig` read it. The two typed Config classes stay — they preserve the module boundary and just resolve the same DSN; the number of configs is not the number of databases.
- **Boot owns migrations for both schemas.** `build_app_from_env` applies the Content Graph and consumption migrations against the one database before serving, so cold start is `createdb` → run app. Seeding the Topic taxonomy stays a manual step — it is content data, not schema.
- **Docker/dev is one database (`rabbithole`)**; `docker-compose` creates it, and there is no second `createdb`.
- **Test isolation is deliberately left alone.** The contract suites keep their own throwaway databases (`CONTENT_GRAPH_TEST_DSN`, `CONSUMPTION_TEST_DSN`): test isolation is orthogonal to production topology, and unifying them would buy nothing.
- **A future physical split is a config change, accepted as YAGNI now.** If content and reader state ever need separate databases for independent scaling, the two Config classes can read distinct vars again — a connection-config change, still not a domain change, exactly as [ADR 0006](0006-generation-and-consumption-are-separate.md) and [ADR 0014](0014-harness-code-lives-in-src-harness.md) intend.
