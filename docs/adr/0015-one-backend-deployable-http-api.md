# The backend is one deployable: one HTTP API serves readers and admin; generation runs in-process and async

The consumption PRD said "backend + its API" but only ever issued in-process application-service classes, and the generation harness is already built as its own package. Now that a real client (mobile + web) must call in, this ADR settles the runtime shape: **the whole backend is one deployable** — one uv project, one venv, one Postgres — exposing **one FastAPI application** with two routers, a **reader router** (the consumption use-cases) and an **admin router** (triggers a generation run). Readers get **anonymous, device-issued identity** (an opaque `user_id` + token minted on first contact; no accounts in V1). Generation is **triggered over HTTP but runs in-process as a background task** — the admin route dispatches the run and returns immediately, so a multi-minute agentic run never blocks reader traffic.

**Why:**
- **Simplicity wins before there is a single user.** One process, one deploy, one test suite — no service mesh, no network hop between "content-graph", "harness", and "api", no independent-deploy machinery to maintain.
- **The boundary is already free.** [ADR 0006](0006-generation-and-consumption-are-separate.md)'s separation is enforced by **import direction and the port surface, not process or repository separation** ([ADR 0014](0014-harness-code-lives-in-src-harness.md)). So co-hosting generation and consumption in one deployable costs nothing at the boundary — the reader module and the generation module simply never import each other.
- **Async trigger, not sync.** A generation run is minutes of LLM calls; running it inside the request path would starve the workers serving readers. Background dispatch keeps everything in one process without wedging generation into the request path.

**Consequence:**
- The **HTTP API layer is in scope** for the consumption work — it was implied by "its API" but never issued. New deliverables: a **reader HTTP router + response DTOs + anonymous-identity token**, an **admin generation-trigger router** (async), and the **composition root** (`src/api/`) that mounts both over the one Postgres. The FE (mobile + web) that *renders* these responses remains deferred.
- **Distinct outbound ports stay distinct** — `ContentGraphRepository` (Postgres), `LLMPort`, `WebSourcePort`: one interface per external system, all files in the one codebase, not separate services. Collapsing them into a single "God port" would force the reader to depend on generation's LLM/web ports, re-fusing the two worlds ADR 0006 keeps apart. Number of ports ≠ number of services.
- **API responses carry internal vocabulary only.** Branded strings ("Thread", "Tapestry") render client-side from the presentation vocabulary module ([ADR 0001](0001-decouple-internal-and-ui-vocabulary.md)); the wire format never emits them, and never emits generation-only fields (`run_id`, constellation).
- **Anonymous identity is a V1 cost, accepted:** a reader who switches devices or clears storage loses their Tapestry until real accounts are added — a clean later addition, not a rearchitecture.
- **Splitting generation into its own deployment later** (its own scaling/cadence) remains a packaging change, not a domain change (per ADR 0014).
