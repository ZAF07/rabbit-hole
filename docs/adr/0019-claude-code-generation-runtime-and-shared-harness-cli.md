# The Claude Code generation runtime & the shared `harness` CLI seam

[ADR 0010](0010-content-generation-pipeline-architecture.md) declared a **dual runtime** — a deterministic LangGraph `StateGraph` for production and *Claude Code skills/agents* for interactive / dev / human-in-the-loop — sharing one markdown-and-manifest source of truth. Only the production runtime was built ([ADR 0016](0016-production-llm-adapter-and-bounded-worker-agents.md)); the Claude Code side had the seven subagent cards but **no conductor**, and their `tools:` grants named a port (`WebSourcePort`) that is not a real Claude Code tool — so it could not actually run. This ADR settles how the second runtime is made real, in a way that keeps "the same run, via Claude Code" a *provable* claim rather than a hopeful one: **Claude drives the creative stages; a shared `harness` CLI enforces the contract and does the write, so both runtimes are held to identical outcomes.**

## Decisions

1. **Claude Code is a second real engine, triggered by a `/new-constellation` skill.** Following the we-os `/new-campaign` pattern, a `.claude/skills/new-constellation/SKILL.md` makes Claude Code the *conductor* of one generation run: it delegates each stage to the existing subagent cards (Architect … Reviewer), Claude itself doing the researching, writing, and judging. This runs **alongside**, not instead of, the production path — the DeepSeek/LangGraph engine behind the admin HTTP trigger ([ADR 0015](0015-one-backend-deployable-http-api.md)) is untouched. The two engines never share an execution path; they share the spec, the manifest, and the run workspace. `/new-constellation` is the interactive runtime; the HTTP trigger is production. DeepSeek is *not* called on the Claude path.

2. **A first-class `harness` CLI (`src/harness/cli.py`) is the shared deterministic core.** Both the human's in-process driver and the Claude subagents invoke it via `Bash`. It exposes exactly the machinery that must be *identical* across runtimes: `check-piece` / `check-constellation` (the pure `evaluate_piece` / `evaluate_constellation` / `evaluate_connections` functions — the binary **Tier-1 invariants I1–I8**, anti-slop phrase bans, grounding-ledger validity), `fetch` (the Playwright `WebSourcePort` recall-first fetch + citation-chase), `verdict` (append to `feedback/verdicts.jsonl`), and `publish` (the Stage-9 rewire → reqa → atomic Content Graph write). It is tested and `mypy`-clean like the rest of `src/harness/`; the scratch `scripts/gen.py` driver is folded into it.

3. **The skill conducts by reading `manifest.toml` at runtime, not by restating the pipeline.** The SKILL.md is a thin conductor persona: at run start it reads [`harness/manifest.toml`](../../harness/manifest.toml) and the agent cards, then walks stages in manifest order, honoring the **deliverable-on-disk prerequisite gate** and firing the three human gates where the manifest declares them. The pipeline is therefore *not* written down twice — edit the manifest and both runtimes follow. No new orchestrator agent card is introduced; the skill body is the conductor (the eighth role we-os folds into its skill, not a subagent).

4. **Claude proposes; the shared core disposes — parity is by construction.** The division of labor mirrors [ADR 0016](0016-production-llm-adapter-and-bounded-worker-agents.md) decision 3 ("the agent proposes; code disposes"), lifted from *inside a stage* to *across the runtime*: the Claude subagents do the creative and judgment work (plan, recall + source, draft, prose/anti-slop revision, hooks, Tier-2 coherence), but every **binary contract check, the grounding-ledger validation, the verdict log, and the DB write are the same Python called through the CLI seam**. Because the Claude runtime is *forced through* the identical gates and the identical write, it cannot publish an out-of-contract constellation any more than the LangGraph runtime can. This is what "the same" means here: **same pipeline, same contract, same write — not identical prose** (a different model writes it).

5. **The Researcher grounds through the same `WebSourcePort`, honoring [ADR 0011](0011-harness-tools-are-ports-web-sourcing-recall-first.md).** On the Claude path the Researcher recalls candidate URLs from its own memory (recall-first), then fetches and citation-chases by calling `harness fetch` — the Playwright `WebSourcePort`, returning raw text + outlinks and writing `grounding.json` identically to the production Researcher. Claude Code's native `WebSearch` (a search engine, forbidden in V1) and `WebFetch` (summarizes rather than returning raw text/outlinks, breaking closed-book fidelity) are **not** used for grounding. The claim pack is produced by the same code, so the two runtimes' substrates do not diverge.

6. **The three human gates reuse the shared verdict contract; run workspaces are cross-engine interoperable.** Both engines write `harness/runs/<id>/` with the same artifact layout, preserve the same `*.machine.md` copies, and pause at the same three gates (plan → each Piece → wired constellation, [ADR 0013](0013-human-review-surface-is-the-file-workspace.md)). On the Claude path the human decides in-session and Claude records the verdict via `harness verdict`; the Distiller ([ADR 0004](0004-human-ratified-learning-loop.md)) therefore learns from both engines' verdicts uniformly (the `model` field distinguishes them). Because the workspace format is shared, a run is *resumable across engines* — though a single run is expected to stay on one engine for coherence.

7. **The subagent cards' `tools:` grants become real.** The placeholder `tools: WebSourcePort` on `researcher.md` (and the analogous grants the Editor and Reviewer need to call `check-*`) are replaced with a real `Bash` grant scoped to the `harness` CLI. This is regenerated from [`harness/agents/README.md`](../../harness/agents/README.md), and the existing `.claude/agents/` drift test continues to guard it.

8. **Automated parity stays on the two Python runtimes; the Claude runtime is validated by construction.** [ADR 0010](0010-content-generation-pipeline-architecture.md)'s parity test — a fixture Brief asserting contract-satisfying output — continues to hold the LangGraph and manifest runtimes byte-for-contract identical. Claude cannot run in CI (non-deterministic, costly), so the Claude runtime is not parity-tested by execution; its correctness rests on decisions 2–4 — it walks the same manifest, honors the same specs, and is forced through the same shared-core gates and write. The seam's `check-*` and `publish` commands *are* covered by the Python test suite, so the contract they enforce is tested once and shared.

## Why

- **"The same" has to be provable.** The founder's whole bet is that these first ~30 Pieces read as *smart and trustworthy*, not AI slop. A second runtime that only *approximates* the contract would let the interactive path publish things the production path never could. Making the contract and the write a single shared implementation both engines are forced through turns parity from a promise into a property.
- **The moat stays in reviewable artifacts.** The manifest, the DNA, the guardrails, and the agent cards remain the single source of truth; adding a runtime adds a conductor and a CLI, not a second copy of the taste.
- **Ports the we-os discipline the founder trusts.** Two engines over one markdown spec, a Stage-0 gate, deliverable-on-disk gating, ports-and-adapters — exactly the shape of `~/we-os`, now with the deterministic core *shared as code* rather than re-judged, which we-os could not do because its guardrails are LLM-graded rubrics and this repo's Tier-1 is binary code.

## Trade-off

More surface than a launcher would need: a first-class CLI, a conductor skill that must faithfully walk the manifest, and real tool grants on the subagents. The lighter alternative — Claude Code as a *launcher* that shells into the DeepSeek engine — was rejected because it is not a second runtime at all (it cannot author or iterate with Claude as the model), which is the point of the interactive path. Re-judging the guardrails from markdown on the Claude side (pure we-os style) was rejected because it would downgrade the deterministic I1–I8 to soft LLM judgement and let the two runtimes silently disagree on what passes.

## Consequence

The [ADR 0006](0006-generation-and-consumption-are-separate.md) boundary is untouched — the Claude runtime writes only Pieces, Connections, and Topic tags through the same `ContentGraphRepository` write port, and imports nothing from consumption. [ADR 0010](0010-content-generation-pipeline-architecture.md) still governs the pipeline's *shape*; this ADR fills in its *interactive runtime* and introduces the shared `harness` CLI as the parity substrate. This ADR is reopened only by a change to *that* shape — the runtime split, the shared-core boundary (what the CLI enforces vs what Claude decides), or the grounding stance. The operator-facing walkthrough for driving a run lives in [`docs/running-a-generation-run.md`](../running-a-generation-run.md), which will grow a `/new-constellation` section as the runtime lands.
